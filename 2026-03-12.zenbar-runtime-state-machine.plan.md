# 2026-03-12.zenbar-runtime-state-machine.plan.md

## Title

Normalize Zenbar task state machine to match Codex runtime events

## Summary

Zenbar currently maps multiple Codex runtime interaction types into a single `waiting_approval` state. This causes incorrect UI behavior when the runtime requests user input through `item/tool/requestUserInput`.

The goal of this change is to align the Zenbar task lifecycle with Codex runtime semantics by introducing a distinct state for user input requests.

Key idea:

Separate **result approval** from **user input requests**.

Current behavior incorrectly treats both as `waiting_approval`.

Target behavior introduces a new state:

```
waiting_user_input
```

This allows the Web Commander UI to render the correct interaction model.

## Current Behavior

Codex runtime emits events such as:

```
item/tool/applyPatch
item/tool/writeFile
item/tool/commit
item/tool/requestUserInput
```

Zenbar currently maps all tool interactions requiring human involvement into:

```
waiting_approval
```

This causes problems when the runtime asks a question rather than requesting approval.

Example:

```
Approval requested: item/tool/requestUserInput
```

But the UI disables the Approve button because the task is in `plan` mode.

Result:

```
Task state: waiting_approval
Approve button: disabled
User cannot respond
```

This is a state machine design bug.

## Target State Machine

Zenbar tasks should support the following states.

```
queued
starting
running
waiting_user_input
waiting_result_approval
approved
stopped
failed
completed
```

Key additions:

```
waiting_user_input
waiting_result_approval
```

## State Definitions

### queued

Task record created but runtime not started.

### starting

Runtime initialization in progress.

Examples:

```
thread/start
workspace preparation
```

### running

Codex runtime actively executing.

Events include:

```
agent_status
command_executed
test_result
plan updates
```

### waiting_user_input

Runtime explicitly requested user input.

Triggered by:

```
item/tool/requestUserInput
```

UI behavior:

* show question
* show input field
* enable `Send response`
* disable `Approve`

### waiting_result_approval

Runtime produced an action requiring approval.

Triggered by:

```
item/tool/applyPatch
item/tool/writeFile
item/tool/commit
item/tool/deleteFile
```

UI behavior:

```
Approve
Stop
```

### approved

User approved the pending action.

Runtime continues execution.

### stopped

User manually stopped the task.

### failed

Runtime or orchestration error occurred.

Examples:

```
runtime connection lost
workspace error
protocol failure
```

### completed

Task execution finished successfully.

For plan tasks this means:

```
plan generation finished
```

No approval required.

## Event Mapping

Codex runtime events should map to Zenbar states as follows.

| Runtime event              | Zenbar state            |
| -------------------------- | ----------------------- |
| thread started             | running                 |
| command execution          | running                 |
| plan update                | running                 |
| item/tool/requestUserInput | waiting_user_input      |
| item/tool/applyPatch       | waiting_result_approval |
| item/tool/writeFile        | waiting_result_approval |
| runtime finished           | completed               |
| runtime error              | failed                  |

## Runtime Adapter Changes

Update the Codex runtime adapter to detect tool invocation types.

Pseudo logic:

```
if tool_call.name == "requestUserInput":
    task.status = "waiting_user_input"

elif tool_call.name in [
    "applyPatch",
    "writeFile",
    "commit",
    "deleteFile"
]:
    task.status = "waiting_result_approval"
```

The adapter should include the tool payload in the event stream.

## Web Commander UI Changes

Update UI behavior based on the new state.

### waiting_user_input

Render:

```
Agent question
Text input
Send response button
Stop button
```

Disable:

```
Approve
```

### waiting_result_approval

Render:

```
Approve
Stop
```

No input field.

### running

Render event stream only.

### completed

Render:

```
event log
diff summary
```

No interaction buttons.

## API Changes

Task schema should support the new states.

Example:

```ts
export type TaskStatus =
  | "queued"
  | "starting"
  | "running"
  | "waiting_user_input"
  | "waiting_result_approval"
  | "approved"
  | "stopped"
  | "failed"
  | "completed";
```

Existing tasks using `waiting_approval` should be migrated to:

```
waiting_result_approval
```

## Backward Compatibility

If existing events emit:

```
waiting_approval
```

Map them to:

```
waiting_result_approval
```

until the runtime adapter is updated.

## Non Goals

This change does not modify:

* Codex runtime behavior
* event streaming protocol
* workspace lifecycle

Only Zenbar orchestration state mapping.

## Expected Outcome

After this change:

Zenbar will correctly support both:

```
approval workflow
```

and

```
agent question / human feedback workflow
```

This aligns the control plane with Codex runtime semantics.
